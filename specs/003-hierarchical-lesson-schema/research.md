# Research: Hierarchical Lesson Plan Schema

**Feature**: 003-hierarchical-lesson-schema
**Date**: 2026-03-19

## Findings

### 1. Drizzle ORM Relational Queries (N+1 Prevention)

**Decision**: Use `db.query.<table>.findFirst({ with: { sections: { with: { items: true } } } })`
for all hierarchy fetches.

**Rationale**: Drizzle's relational query API (`drizzle-orm` `with` operator) performs
a single SQL query with JOINs or a minimal set of batched queries — it does NOT
produce N+1. The `relations()` helper in `schema.ts` must define the
`lesson_plans → lesson_sections → lesson_items` relationships. Drizzle `with`
respects `orderBy` at each level, so ordering by `display_order ASC` is achievable
in a single fetch.

**Alternatives considered**:
- Raw SQL JOIN: More control but bypasses type safety and Drizzle's inference.
- Sequential `.select()` calls: Produces N+1. Rejected.

---

### 2. Slug & Anchor ID Generation

**Decision**: Implement a shared `slugify(input: string): string` utility in
`src/lib/utils/slugify.ts`. Rules: lowercase, trim, replace non-alphanumeric
characters with hyphens, collapse consecutive hyphens, strip leading/trailing hyphens.

**Rationale**: Slugs must match the heading IDs generated by `react-markdown` with
`remark-gfm` (already a project dependency). GitHub-Flavored Markdown heading IDs
follow exactly this pattern. Matching the renderer's output ensures `/#anchor`
navigation works without a client-side mapping table.

**Example**: `"Required Tools & Technologies"` → `"required-tools-technologies"`

**Alternatives considered**:
- `github-slugger` library: Accurate but adds a dependency; not needed since the
  logic is simple enough to inline and test.
- Store explicit anchor IDs separately from titles: Allows decoupling but adds
  synchronization complexity. Rejected — auto-derivation from title is sufficient
  and simpler to keep consistent.

---

### 3. Migration Strategy for Existing `content: jsonb` Data

**Decision**: Two-step migration:
1. Add `lesson_sections` and `lesson_items` tables.
2. For each `lesson_plans` row where `content != '{}'::jsonb`: insert a
   `lesson_sections` row with `title = 'Content'`, `slug = 'content'`,
   `display_order = 1`, `content_markdown = content::text`.
3. Drop `content` column from `lesson_plans`.

Steps 1–3 are combined in a single `drizzle-kit generate` migration SQL file
using a `DO $$ ... $$` migration block for the data transfer.

**Rationale**: No data loss. The migration is reversible (in dev) by recreating
the column and back-populating. In production, a single atomic migration transaction
prevents partial state.

**Alternatives considered**:
- Keep `content` as nullable fallback: Creates schema drift and requires dual-path
  reads. Rejected.
- Manual migration script outside drizzle: Harder to track and replay. Rejected.

---

### 4. Unique Constraint Enforcement

**Decision**: Database-level UNIQUE constraints for `(lesson_plan_id, slug)` on
`lesson_sections` and `(section_id, anchor_id)` on `lesson_items`. Zod validates
format at the API layer before hitting the DB; the DB constraint is the final
safety net.

**Rationale**: Application-level uniqueness checks (read-then-write) are racy under
concurrent requests. The DB constraint is atomic.

**Error handling**: When a unique constraint violation is caught (Postgres error code
`23505`), the API route returns HTTP 409 with a descriptive JSON error body:
`{ "error": "slug_conflict", "message": "A section with slug '...' already exists in this plan." }`

---

### 5. Display Order Strategy

**Decision**: Store display order as a sparse integer (gaps allowed). Default
to `MAX(display_order) + 10` when appending. Reordering updates `display_order`
values in a single bulk UPDATE.

**Rationale**: Sparse integers (10, 20, 30…) allow inserting between existing items
without renumbering. A full renumber is only needed if the gap runs out (unlikely
in practice). No fractional ordering (e.g., Jira-style) needed at this scale.

**Alternatives considered**:
- Linked list (prev_id/next_id): Complex to query and reorder. Rejected.
- Float ordering: Precision issues over time. Rejected.

---

### 6. Testing Infrastructure

**Decision**: Add `vitest` and `@vitest/coverage-v8` to devDependencies for unit
testing the `slugify` utility and any other pure functions. Playwright (already
installed) covers integration and e2e tests against a local Supabase instance.

**Rationale**: The constitution requires unit tests for business logic. The `slugify`
utility is pure (no I/O), making it ideal for fast unit tests. Integration tests via
Playwright handle the full API contract (auth, DB round-trip, constraint enforcement).

**Alternatives considered**:
- Jest: Heavier setup for an ESM + TypeScript project. Vitest is a drop-in
  replacement with native ESM and TypeScript support.
