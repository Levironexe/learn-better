# Implementation Plan: Hierarchical Lesson Plan Schema

**Branch**: `003-hierarchical-lesson-schema` | **Date**: 2026-03-19 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/003-hierarchical-lesson-schema/spec.md`

## Summary

Restructure the lesson plan data model from a flat single-table design (with a
`content: jsonb` blob) into a three-level hierarchy: **Lesson Plan → Sections →
Items**. Sections hold the full markdown content; items are anchor references
within their parent section's markdown. Chat messages continue to reference
lesson plans unchanged. The change requires a Drizzle schema update, a
data migration for any existing content, new REST API routes for sections and
items, and updated API routes that return the full hierarchy.

## Technical Context

**Language/Version**: TypeScript 5.9 + Node.js 20+
**Primary Dependencies**: Next.js 16.1.7, Drizzle ORM 0.45.1, drizzle-kit 0.31.10,
  postgres 3.4.8, Supabase Auth (@supabase/ssr 0.9.0), Zod 4.3.6
**Storage**: PostgreSQL via Supabase (pooled connection uses `DATABASE_URL`;
  migrations use `DATABASE_URL_UNPOOLED`)
**Testing**: Playwright 1.58.2 (integration/e2e); vitest recommended for unit tests
  on slug/anchor utilities (not currently in devDependencies — add it)
**Target Platform**: Vercel / production Next.js deployment
**Project Type**: Full-stack web application (Next.js App Router, API routes)
**Performance Goals**: Full hierarchy load for a lesson plan (5 sections, 20 items)
  in a single query; page navigation and anchor scroll ≤1 second
**Constraints**: Supabase PgBouncer requires `prepare: false` on the postgres
  client (already set). Auth MUST be verified on every mutating route.
**Scale/Scope**: Per-user data; moderate load. Indexes on FK+order columns are
  sufficient for expected scale.
